# 品質管理とは

## 品質とは

- ISO9000:2015
  - 「対象は本来備わっている特性のあつまりが、要求事項を満たす程度」
- JIS Z 8010:1981
  - 「品物またはサービスが使用目的を満たしているかどうかを決定するための評価の対象となる固有の性質・性能の全体」
- 上記をひらたくいうと、
  - 「何かを利用したときに、役に立つ・うれしいと感じるポイント」のこと

## 品質がよいということ
- 「役に立つ・うれしいと感じるポイント」は、多くの人が共通に認識するものが多くある
  - 例えば、食事の品質を考えると、美味しい、適度に量があるというところは共感を集めるところである。
- その時の事情や、個人の好みで別の品質がより優先して求められることがある。
  - 急いでいるときは、食事が速く提供されるとうれしい。
  - 激辛メニューは、好んで選ぶ人と敬遠する人が分かれる。

## 品質が安定しないと
- 日によって品質がバラバラというのは困る。
  - 今日、あの店で美味しいランチ食べよう、って行ったのに、失敗作に当たってしまったら悲しい。
- とはいえ、人間にいつも完璧を求めるのは無理。
  - その日の調子によって、ちょっとだけ味が変わってしまうこともある。
  - 素材も日によってちょっとずつ違うので、同じ調理でも味が違うかもしれない。

## 品質を保つために
- 品質を保つためには、品質をチェックして必要に応じて補正する手順を組み込まなければならない。
  - 調味料や加熱時間などを数値化して、いつも同じにする。
  - 調理器具をチェックして、切れ味や火力を整える。
  - 食材の状態をチェックして、調味料や加熱時間を微調整する。
  - 味見をして、提供して大丈夫かどうか確認する。
- この、品質をチェックして補正する手順が「品質管理」である。
  - Quality Control、略してQCと言う。
  - 参考) 品質管理は、モノを提供するまでの自分の作業の中で行う。対して、品質保証は、提供されたモノの維持やアフターフォローを含めた活動を指す。

## 品質チェック
- いつ、何をチェックするかを明確することが重要
  - 投入する食材や調味料の分量、火加減や加熱時間など、チェックをすることで効果を出しやすい作業を見つけ出すことができる。
  - それぞれ、チェックするタイミングが限定される。調味料を投入した後に、調味料の分量を確認しても手遅れである。
- 複数の人で共同作業をする場合は、明文化するのも有効は方法です。
  - レシピには、食材の分量とか、火力・加熱時間など書かれている。

## 品質管理のいろいろ
- 提供するモノに直接関わる部分は、品質に直結するので、管理する必要がある。
  - 材料・器具・調理手順などは直接に料理に関わる
- 提供するモノに直接関わらなくても、業務として実施しているものであれば、最終的に提供物と何らかの関係が出てくる。
  - 予約や問い合わせの対応、従業員の管理、収支計算など。
- SDGsなどの社会的要求に対する品質を管理するケースも増えている。

## 品質を考えてみよう！
- 来週の金曜日に近くのお店を予約して、友人とちょっとしたパーティーを開催します。
- あなたのおすすめのお店を予約したことにして、10行くらい案内文を作成してください。
- この作業の中で、どのような品質管理が必要ですか？

- 【解答のチェックポイント】
  - 日時と場所は明示されているか？
  - 参加表明の方法や締め切りは書かれているか？
  - 当日の連絡先は示されているか？
  - 参加費用は書かれているか？
  - 料理の情報、パーティー中のイベント、ドレスコードなど、案内文を読む人の目的や期待にそった情報が書かれているか？

# 情報システムの品質管理

## 情報システム品質
- 情報システムの品質を考えてみよう。
- 情報システムで、「役に立つ・うれしいと感じるポイント」は何ですか？

- 以下、一例です。
  - バグがなく想定通りに動くこと。    
  想定外の異常終了などがないこと。
  - 想定外のサービス停止がないこと   
  アクセス集中による利用制限が限定的であること
  - 目的の達成のために情報システムが効果的効率的に使えること。   
  システムの実用性や快適性で満足感が得られること。
  - リーズナブルなコストで使えること。

- 現在は、QCDSのバランスが重要と考えられている。以前では狭義の品質のみを考えていた。
  - Q (Quality) 狭義の品質(バグが少ない)
  - C (Cost) 費用
  - D (Delivery) 納期
  - S (Scope) 機能範囲

## 情報システムの品質管理
- 情報システムの品質についても、高い状態で維持することが望まれる。
- そこで、品質管理が必要であるが、情報システムならではの特性がある。
  - 成果物の提供に至るまでの期間が長期になりやすい
  - 人がドキュメントやソースコードを書く作業が圧倒的に多い
  - 同じ作業を繰り返す部分は限定されていて、入力や作業内容が変わってくる部分が多い
  - 成果物を提供した後も、運用で作業が継続することが多い
    - 個人が頭を使って実施する作業なので、人によるゆらぎや人的ミスが入り込みやすい

## 品質管理のアプローチ
- 最終的な品質に向かって、工程途中の品質を評価・改善を実施する。
- 人によるゆらぎをなくこと＝作業品質の管理
  - 工程の標準化、WBS(Work Breakdown Structure)への分解、WBSで実施する内容の明確化
  - WBSに必要なインプット、実施スキル、工数の明確化
- 人的ミスを残さないようにすること＝成果物品質の管理
  - ドキュメントやソースコードのレビュー
  - プログラム・システムのテスト

- 求められる品質を作り込んでいる工程の例です。

| 求められる品質                                                               | その品質を作りこむ代表的工程                   |
| :--------------------------------------------------------------------------- | :--------------------------------------------- |
| バグがなく想定通りに動くこと。想定外の異常終了などがないこと                 | 内部設計、コーディング                         |
| 予定外のサービス停止がないこと。アクセス集中による利用制限が限定的           | インフラ構築、デプロイ、運用                   |
| 目的達成のため効果的効率的に使えること。実用性や快適性で満足感が得られること | 要件定義、外部仕様設計                         |
| リーズナブルなコストで使えること                                             | 日時作業やバックオフィスを含めた、あらゆる作業 |

# 品質管理の運用

## 品質管理を行うために
- 品質管理を作業プロセス組みこむことは重要
- 品質管理を確実に実行することを考えましょう。
  - 品質管理プロセスを実施したことの確認
  - 品質管理プロセスで確認する内容の明確化
 
## 品質管理プロセスの確認
- 品質確認プロセスを実施したことの確認は、多くの場合、作業報告書が提出されていることで確認できるでしょう。
 - 例えば、レビュー報告書やテスト結果報告書が該当する。
 - 報告書の形を取らなくても、業務報告などに記載することも多い。
- 

## 品質管理で確認するもの
- 品質確認で確認する対象は工程によってさまざまです。
- 多くの場合は、中間生産物の品質を評価する。
  - 品質確保作業に費やした作業量を評価する場合もある。
- 評価の指標はさまざまです。
  - レビューやテストであれば、作業量(仕様書ページ数・ソースコード行数)あたりの検出バグ数がよく使われる。
  - 中間生産物が数値評価しにくいものは、作業評価になりがちです。

## 品質確認のポイント
- 人が品質をチェックする場合には、品質をチェックする作業にも人的ミスが紛れ込む可能性がある。
  - 人が書いたドキュメントを、人がレビューするときに、書いた人とレビューした人が同じ間違いをしてしまうと、そのバグは検出されない。
  - 特に、書き漏らしはレビューでもスルーされてしまうことがある。
- こういったことを防止するために、何をどうチェックするという内容をリスト化して、チェックリストを作成することが行われている。
  - 特に、前に同内容のチェックを実施してから時間が開く場合は、リマインドの効果も期待できる。

## 品質チェックのポイント(例)
- 指定の仕様書フォーマットを利用していること
- 「です」調、「である」調が、指定の基準に従っていること
- 指定の用語が、指定の意味に則って使われていること
- 各文の表現範囲を限定し、短く簡潔に記述されていること
- 業務的なルールが、表などを使ってMECEに表現されていること
  - Mutually Exclusive and Collectively Exhaustive = 漏れなく、重複なく
- 前工程仕様書との紐づけが明示され、追跡できること
- ドキュメント体系が整理され、指定のIDで認識できること

## チェックリストの留意点
- チェックリストが膨大にならないようにする
  - 一度にチェックできるよう内容は量的に限定される。
  - 複数のステップに確認の観点を分けて、順次適用するようなことを考えましょう
- チェックリストに書いていることがすべてではないと意識しましょう。
  - 確認の観点の範囲で、チェックリスト以外の問題がないかを意識しましょう。
- チェックリストは適切な周期でメンテナンスしましょう。

## 自己品質管理のススメ
- 個人の作業で、よく作業自体を忘れたり、他者から誤りを指摘をされたりすることがある。
- そんなときは、自己品質管理のチェックリストを作ってみては如何でしょうか？
  - 何か課題があったら、次に同じことを繰り返さないようにチェックリストにのせる。
  - 毎日、リストにのせた品質管理ができていることを確認する
  - ひと月くらい実施して、大丈夫だと思ったら、チェックリストから外す。
